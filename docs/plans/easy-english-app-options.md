## 결정 1 — 플랫폼

| | A. 웹앱 (React + Vite) | B. 네이티브 앱 (React Native) | C. PWA |
|--|--|--|--|
| 배포 속도 | ✅ 즉시 | ❌ 스토어 심사 필요 | ✅ 즉시 |
| 설치 부담 | ✅ 없음 (URL 접속) | ❌ 앱 설치 필요 | 🔺 설치 선택 가능 |
| 오프라인 | ❌ 미지원 | ✅ 지원 | 🔺 부분 지원 |
| 음성(TTS) | ✅ Web Speech API | ✅ 네이티브 TTS | ✅ Web Speech API |
| MVP 적합성 | ✅ 가장 빠름 | ❌ 개발 비용 높음 | 🔺 설정 복잡 |
| **선택** | ✅ **채택** | | |

**선택 근거:** MVP 목표는 빠른 검증. 웹앱은 URL 하나로 바로 접속 가능해 초등 저학년·노년층 타깃의 "앱 설치" 장벽을 제거한다. 오프라인은 MVP 범위 외이므로 네이티브의 장점이 현재 단계에서 의미 없음.

## 결정 2 — 학습 방식

브레인스토밍 시각 자료(Visual Companion)에서 3가지 방식을 비교 검토했다.

| | A. 플래시카드 | B. 4지선다만 | C. 단계별 (보기→퀴즈→발음) |
|--|--|--|--|
| 학습 깊이 | 🔺 얕음 (자기평가) | 🔺 중간 | ✅ 깊음 |
| 시간 | ✅ 가장 짧음 | ✅ 짧음 | 🔺 단어당 더 걸림 |
| 기억 효과 | 🔺 수동적 노출 | 🔺 퀴즈만으론 부족 | ✅ 보고→맞추고→듣기 반복 |
| 초보자 부담 | ✅ 없음 | ✅ 낮음 | 🔺 단계 이동이 낯설 수 있음 |
| 발음 포함 | ❌ 별도 구성 필요 | ❌ 별도 구성 필요 | ✅ 자연스럽게 내장 |
| **선택** | | | ✅ **채택** |

**선택 근거:** 타깃(초등 저학년·노년층)은 "한 번 보고 끝"이 아니라 반복 노출이 효과적이다. 3단계 구조는 한 단어를 눈→손→귀로 세 번 접하게 해 기억 정착률이 높다. 단계별 진행이 오히려 "여기까지 했다"는 성취감을 준다.

## 결정 3 — UI 톤 & 분위기

브레인스토밍 시각 자료에서 두 가지 스타일과 혼합안을 검토했다.

| | A. 귀엽고 친근한 스타일 | B. 깔끔하고 큰 글씨 스타일 | C. 혼합 |
|--|--|--|--|
| 아동 반응 | ✅ 흥미 높음 | 🔺 덜 흥미로울 수 있음 | ✅ 균형 |
| 노년층 반응 | ❌ "장난감 같다" | ✅ 신뢰감, 가독성 | ✅ 균형 |
| 개발 복잡도 | 🔺 캐릭터 리소스 필요 | ✅ 단순 | 🔺 두 스타일 관리 |
| 브랜드 일관성 | ❌ 타깃 충돌 | ✅ 두 페르소나 모두 수용 | ✅ |
| **선택** | | ✅ **채택 (기본)** | 🔺 이모지 활용으로 부분 적용 |

**선택 근거:** 두 페르소나(아동·노년층)를 동시에 수용해야 하므로 B 스타일 기본 적용. 단, 이모지(🍎 🔊 🔥)를 적극 활용해 시각적 친근함을 A 수준으로 보완한다. 별도 캐릭터 리소스 없이도 충분히 생동감 있는 UI 구현 가능.

## 결정 4 — 진도 저장 방식

| | A. 로그인 + 서버 저장 | B. localStorage (비로그인) | C. 기기 저장 (네이티브) |
|--|--|--|--|
| 구현 속도 | ❌ 인증 시스템 필요 | ✅ 즉시 | — (웹 제외) |
| 사용자 마찰 | ❌ 가입 장벽 | ✅ 없음 | — |
| 데이터 영속성 | ✅ 기기 바꿔도 유지 | 🔺 브라우저 삭제 시 소실 | — |
| MVP 적합성 | ❌ 과도한 범위 | ✅ 최소 구현 | — |
| **선택** | | ✅ **채택** | |

**선택 근거:** 타깃 사용자에게 "가입하세요"는 최대 이탈 지점이다. localStorage로 즉시 시작하게 하고, 데이터 소실 리스크는 MVP 단계에서는 수용한다. Phase 2에서 선택적 동기화 검토.

## 최종 결정 요약

| 결정 항목 | 선택 | 핵심 이유 |
|----------|------|----------|
